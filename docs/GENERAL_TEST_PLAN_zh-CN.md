# Shinsekai 通用测试计划

本文用于 RC、发布前验收和高风险功能变更的人工测试。执行时优先覆盖主路径；如果某项不适用于当前版本，需要在 release tracking issue 或 PR test plan 中说明原因。

## 测试准备

- 使用干净用户数据目录，模拟首次安装用户。
- 准备一套可用的 LLM API 配置，至少能完成一次普通文本回复。
- 准备测试资源：角色包 `.char`、背景包 `.bg`、特效包 `.ef`、至少一张立绘图片、至少一张背景图片、至少一个短音频文件。
- 记录测试环境：操作系统与架构、安装包类型与版本号、是否首次安装、是否启用本地 TTS/ComfyUI/插件/代理。

## 1. 新用户最小设置

目标：确认新用户从空配置到启动第一场聊天的最短路径可用。

- 首次启动应用。
- 进入 API 设置。
- 配置 LLM 服务商、Base URL、API Key 和模型 ID。
- 点击模型列表获取或连通检测。
- 保存 API 设置。
- 关闭并重新打开应用，确认设置仍然存在。
- 若本地 runtime 缺失，确认提示清晰，安装/更新流程可执行，失败时有可读错误信息。

通过标准：

- 新用户不需要手动编辑配置文件即可完成最小设置。
- 保存后的 API 设置在重启后仍然生效。
- 连通失败时能看到明确错误，不出现空白页或无响应。

## 2. 角色包导入、编辑与导出

目标：确认角色资源可导入、可编辑、可导出，并能参与聊天。

- 导入一个 `.char` 角色包。
- 确认角色出现在角色列表。
- 打开角色详情，检查名称、简介、设定、颜色、立绘、情绪标签等字段。
- 修改角色名称或设定并保存。
- 上传一张新的立绘。
- 调整立绘缩放。
- 为立绘设置情绪标签。
- 可选：为立绘绑定一段台词音频。
- 导出角色包。
- 删除该角色。
- 重新导入刚导出的 `.char`。

通过标准：

- 导入后字段完整，资源路径有效。
- 编辑保存后刷新页面或重启应用仍然存在。
- 导出的 `.char` 能重新导入，且关键字段和资源未丢失。
- 导入同名角色时有可理解的处理方式或错误提示。

## 3. 背景包导入、编辑与导出

目标：确认背景组和 BGM 资源可完整往返。

- 导入一个 `.bg` 背景包。
- 确认背景组出现在背景列表。
- 检查背景名称、资源目录、图片标签、BGM 标签。
- 上传一张背景图片。
- 上传一段 BGM。
- 修改图片标签和 BGM 标签并保存。
- 导出背景包。
- 删除该背景组。
- 重新导入刚导出的 `.bg`。

通过标准：

- 背景图片和 BGM 能正常显示或试听。
- 导出再导入后，背景组、标签和资源仍然完整。
- 删除背景组后列表和配置状态正确刷新。

## 4. 特效包导入、编辑与导出

目标：确认特效方案可导入、编辑、导出，并能被聊天模板引用。

- 导入一个 `.ef` 特效包。
- 确认特效方案出现在特效列表。
- 检查特效名称、颜色、音频列表和音频标签。
- 上传一段特效音频。
- 修改特效音频标签并保存。
- 导出特效包。
- 删除该特效方案。
- 重新导入刚导出的 `.ef`。

通过标准：

- 特效方案导入后字段完整。
- 特效音频能正常保存和播放。
- 导出的 `.ef` 能重新导入，且音频与标签不丢失。

## 5. 聊天模板与启动聊天

目标：确认角色、背景、特效能够组合成可启动的聊天模板。

- 进入聊天模板页面。
- 选择至少一个角色。
- 选择一个背景组。
- 启用或关闭特效，分别生成一次模板。
- 保存模板。
- 使用该模板启动聊天。
- 确认聊天窗口打开。
- 发送一条用户消息。
- 等待模型回复。
- 确认角色名、文本、立绘、背景和可选特效显示正常。

通过标准：

- 模板生成和保存成功。
- 启动聊天不报错、不白屏。
- 至少完成一轮用户消息与模型回复。
- 角色、背景、特效开关与模板配置一致。

## 6. 聊天记录保存与恢复

目标：确认会话历史能保存、恢复，并可继续聊天。

- 启动一次聊天。
- 发送至少两条用户消息，并等待回复。
- 保存聊天记录。
- 关闭聊天窗口。
- 返回设置界面。
- 使用“加载上次聊天并启动”或等价恢复入口。
- 确认恢复后的聊天窗口包含上一轮消息。
- 在恢复后的会话中继续发送新消息。
- 再次保存聊天记录。

通过标准：

- 聊天记录文件生成成功。
- 恢复后消息顺序、角色名和正文内容正确。
- 恢复后的会话可以继续发送消息，不会重置上下文。
- 保存失败时有可读错误提示。

## 7. 配置持久化与重启恢复

目标：确认用户配置不会在重启后丢失。

- 修改 API 设置并保存。
- 修改系统设置，例如语言、深色模式或代理配置。
- 修改角色、背景、特效中的至少一项。
- 完全退出应用。
- 重新启动应用。
- 检查上述配置是否仍然存在。
- 再次启动聊天，确认配置仍被使用。

通过标准：

- 重启后配置完整保留。
- 页面加载过程中没有长时间空白或错误状态。
- 配置读取失败时能提示具体问题。

## 8. 插件与工具基础检查

目标：确认插件管理和工具调用没有破坏主流程。

- 打开插件管理页面。
- 查看已安装插件列表。
- 启用或禁用一个非关键插件。
- 重启应用。
- 确认插件状态保留。
- 在聊天或工具页面触发一次基础工具调用。

通过标准：

- 插件列表能正常加载。
- 启用/禁用状态重启后仍然正确。
- 插件或工具错误不会导致主应用崩溃。

## 9. 资源异常与错误处理

目标：确认常见错误不会造成崩溃或数据损坏。

- 尝试导入格式错误的 `.char`、`.bg` 或 `.ef` 文件。
- 尝试导入不存在或损坏的资源文件。
- 尝试导入同名角色、背景或特效。
- 在 API Key 错误的情况下启动聊天。
- 在网络不可用的情况下获取模型列表。
- 在缺少 runtime 依赖的情况下启动聊天。

通过标准：

- 应用不崩溃。
- 错误提示能说明失败原因。
- 已有角色、背景、特效和配置不被破坏。
- 修正错误后可以继续操作。

## 10. 更新与发布包检查

目标：确认发布包和更新信息可用。

- 检查应用显示版本号是否与 release tag 一致。
- 检查安装包能正常安装或覆盖安装。
- 检查首次启动和二次启动都正常。
- 检查 release assets 可下载。
- 检查 `latest.json` 存在且版本、平台、签名字段正确。
- 若支持自动更新，确认更新提示能出现，取消/关闭后应用状态正常。

通过标准：

- 安装包可安装、可启动、可再次启动。
- 版本号与发布版本一致。
- updater manifest 和签名字段完整。

## 11. 回归范围选择

每次 release 至少执行第 1、5、6、10 节。

如果本次版本包含相关改动，需要追加执行：

- 改角色管理：执行第 2 节。
- 改背景管理：执行第 3 节。
- 改特效管理：执行第 4 节。
- 改模板或聊天启动：执行第 5、6 节。
- 改配置、runtime、代理、语言：执行第 1、7、9 节。
- 改插件或工具：执行第 8 节。
- 改安装包、updater、版本号：执行第 10 节。
